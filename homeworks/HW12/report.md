# HW12 - временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- Использовался датасет `S12-hw-dataset.csv`; прогнозировалась колонка `target` на 1 шаг вперед (`horizon=1`).
- Данные отсортированы по времени, затем выполнен строгий `train / validation / test` split по хронологии.
- Сравнены четыре обязательных подхода: `B1 (naive-last)`, `B2 (moving-average)`, `B3 (ridge-lag-features)`, `R1 (gru-forecast)`.
- Лучшим по валидации оказался `R1 (GRU)` с минимальным `val_MAE`; именно он финально оценен на test.

## 3. Данные и постановка задачи

- Датасет: `S12-hw-dataset.csv`
- Целевая колонка: `target`
- Частота временного ряда: почасовая
- Горизонт прогноза: `1`
- Размер датасета: `4320`
- Диапазон дат: `2025-01-01` - `2025-06-29`
- Есть ли дополнительные внешние признаки: нет
- Комментарий (2-4 предложения): Ряд демонстрирует выраженную сезонность и локальные колебания, что важно для выбора окна и признаков. Пропуски в исходных данных отсутствуют. Для корректной оценки прогноза ключевым является сохранение хронологии: модель должна учиться на прошлом и проверяться на будущем. Поэтому random split в основной постановке не используется.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- Как именно выполнен `train / validation / test`: последовательный разрез после сортировки по `date`.
- Какие доли или интервалы использовались: `70% / 15% / 15%`.
- Почему `random split` здесь некорректен: он смешивает прошлое и будущее и дает нереалистично оптимистичную оценку качества для forecasting-задачи.

### 4.2. Признаки для baseline-моделей

- Какие лаги использовались: `lag_1`, `lag_7`, `lag_14`.
- Какие rolling-признаки использовались: `rolling_mean_7`, `rolling_std_7`.
- Какие календарные признаки использовались: день недели (`day_of_week`).
- Как обрабатывались пропуски после построения лагов/rolling: удалялись первые строки с `NaN`.
- Использовалось ли масштабирование и где именно оно обучалось: `StandardScaler(features)` только на train для `B3`, и `StandardScaler(target)` только на train для `R1`.

## 5. Модели и эксперименты (B1, B2, B3, R1)

Опишите коротко и сопоставимо:

- B1 (`naive-last`): прогноз равен последнему известному значению ряда.
- B2 (`moving-average`): прогноз равен скользящему среднему по окну 7.
- B3 (`ridge-lag-features`): `Ridge` на лаговых, rolling и календарном признаках.
- R1 (`gru-forecast`): GRU на оконном представлении ряда (`window_size=24`).

Дополнительно:

- Основная метрика выбора лучшей модели: `best_val_mae`.
- Дополнительные метрики: `RMSE`, `MAPE`.
- `window_size` для `GRU`: `24`.
- Batch size: `64` (train), `128` (validation/test).
- Максимальное число эпох: `30`.
- Optimizer: `Adam`.
- Learning rate: `0.001`.
- Как сохранялась лучшая модель: `state_dict` в `./artifacts/best_gru.pt`, конфиг в `./artifacts/best_gru_config.json`.

## 7. Анализ

Наивный baseline (`B1`) на этом ряде оказался сильным, что типично для последовательностей с высокой автокорреляцией на коротком горизонте. Подход `B2` со скользящим средним ухудшил качество, так как сглаживание вносит запаздывание и хуже реагирует на локальные изменения уровня. `B3` с лаговыми и rolling-признаками частично компенсирует это за счет большего контекста, но в данной конфигурации не обошел `B1`.

`GRU` (`R1`) показала лучший результат на validation, что говорит о способности лучше извлекать нелинейную динамику из окна длиной 24 шага. При этом разница между validation и test умеренная, без резкого ухудшения, что указывает на приемлемую обобщающую способность. Основные риски утечек в такой задаче: random split, rolling без сдвига, fit scaler на полном датасете. В решении эти риски закрыты: temporal split, признаки только из прошлого, масштабирование только по train.

Выбор лучшей модели именно по validation принципиален: если использовать test для тюнинга, это приводит к завышенной оценке. Наиболее показательные ошибки модели наблюдаются в участках быстрых локальных сдвигов, где даже сильные autoregressive паттерны временно нарушаются.

## 8. Итоговый вывод

Для данной задачи в качестве надежной отправной точки можно брать `B1`, потому что он очень простой и уже дает высокий baseline-уровень. Если нужна лучшая точность, имеет смысл переходить к `GRU`, как показано в `R1`. Признаковые табличные модели полезны как интерпретируемая промежуточная ступень, но не всегда автоматически выигрывают у наивного прогноза. Главный практический вывод HW12: корректная временная валидация важнее «сложности» модели, потому что именно она делает сравнение честным и переносимым на будущее.
